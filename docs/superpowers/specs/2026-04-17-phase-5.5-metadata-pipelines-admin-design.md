# Phase 5.5 — Metadata Pipelines Admin UI

**Date:** 2026-04-17
**Status:** Design — awaiting implementation plan
**Owner:** Phase 5 track (web-ui + control-plane)
**Supersedes:** "Phase 5.5 — Pipeline admin UI (tracked task #13)" open item in `.claude/handoff-spaceharbor.md`

## 1. Problem

The `dataEnginePipelines` platform setting drives which VAST DataEngine function SpaceHarbor invokes for each `fileKind` (`image` / `video` / `raw_camera`) and which VastDB `schema.table` the control-plane reads on the unified metadata endpoint (`GET /assets/:id/metadata`, shipped as C-1b). Today the setting is only editable via raw `PUT /api/v1/platform/settings` calls. Operators cannot:

- See which pipelines are currently configured without piping `curl | jq`.
- Confirm the configured `functionName` actually resolves to a live VAST DataEngine function (or learn *why* it doesn't — `function-not-found` vs. `vast-unreachable`).
- Verify a given `targetSchema.targetTable` actually returns rows for a real asset before saving a change.

This surface is the last gap in Phase 5: every other admin-controlled concept (naming templates, triggers, workflows, plugins) has a dedicated page under `/automation/*`.

## 2. Scope

**In scope**

- New web-ui page at route `/automation/pipelines`, nav label **"Metadata Pipelines"**.
- Table view of the 3 possible pipelines (one per `fileKind`), each row showing: fileKind, function name, `targetSchema.targetTable`, extensions, live status pill, enabled toggle, Edit button.
- Edit dialog with three panes: editable form, read-only live VAST function record, per-pipeline test-lookup tool.
- Empty-state seed button that populates the list from the container's seed file in one click.
- "Seed missing" banner + action when the list is non-empty but some `fileKind`s are absent.
- Force-refresh button bypassing the 60 s discovery cache.
- Inline enable/disable toggle per row (optimistic, rolled back on save failure).
- One new control-plane route: `GET /api/v1/dataengine/pipelines/defaults` (reads the same seed JSON as `pipeline-seed.ts`).
- One new control-plane proxy route: `GET /api/v1/metadata/lookup` (thin forwarder to `vastdb-query:8070/api/v1/metadata/lookup`).

**Out of scope**

- Deleting individual pipelines. Toggling `enabled` is the soft-off. Manual external edits may still reduce the list below 3 entries; the "Seed missing" banner is the recovery path.
- Creating a pipeline for a *new* `fileKind` that is not `image`/`video`/`raw_camera` (requires backend schema work).
- Test-dispatch (invoking a VAST function on a user-entered path). Significant backend surface; separate spec if ever picked up.
- Deep-linking to the existing Functions tab.
- Auto-polling live status. Operators visit briefly; force-refresh button is enough.
- Cross-site CORS from web-ui → `vastdb-query` directly. All vastdb-query traffic from web-ui goes through the control-plane proxy.

## 3. Non-goals / explicit decisions

| Decision | Choice | Rationale |
|---|---|---|
| Page structure | Flat table + modal dialog (NamingTemplatesPage style) | Consistency with siblings in `pages/admin/` |
| File layout | Single file, dialog co-located | Matches NamingTemplatesPage (410 lines); ~320 lines estimated |
| Validation authority | Server-only via `validatePipelineConfigList` | Single source of truth; UI renders `InvalidPipelineConfigError` verbatim |
| Default pipeline set source | Shared seed JSON (no duplication) | Respects `feedback_no_hardcoded_values.md`; one source of truth with `pipeline-seed.ts` |
| vastdb-query endpoint | Control-plane proxy | Avoids CORS + web-ui hardcoding vastdb-query URL |
| Polling | Manual refresh only | Low visit frequency; force-refresh bypasses 60s discovery cache |
| Unsaved-changes warning | Not included | Small form; can add if operator feedback asks |

### 4.2 Data flow

```
[ Page mount ]
  GET /api/v1/dataengine/pipelines/active
    → [{config, live, status, statusDetail?}, ...]
  Derive: configuredKinds, missingKinds, isEmpty

[ Refresh button ]
  GET /api/v1/dataengine/pipelines/active?force=true

[ Seed-defaults (empty state) ]
  GET /api/v1/dataengine/pipelines/defaults
    → DataEnginePipelineConfig[]
  PUT /api/v1/platform/settings { dataEnginePipelines: defaults }
  GET /api/v1/dataengine/pipelines/active?force=true

[ Seed-missing (banner) ]
  defaults ← GET /defaults
  merged ← [...existing, ...defaults.filter(d => missingKinds.includes(d.fileKind))]
  PUT /platform/settings { dataEnginePipelines: merged }
  GET /active?force=true

[ Inline toggle ]
  next ← rows.map(toggle target.enabled)
  optimistic setState(next)
  PUT /platform/settings { dataEnginePipelines: next }
    on error → rollback + banner
    on success → GET /active?force=true (picks up live-record re-resolution)

[ Edit dialog save ]
  next ← rows.map(replace target with formConfig)
  PUT /platform/settings { dataEnginePipelines: next }
    on 400 → dialog stays open, renders error envelope message inline
    on 200 → close, GET /active?force=true

[ Test-lookup (inside dialog) ]
  // Uses dialog's CURRENT form values (not saved state) so operators
  // can verify new schema/table before committing.
  GET /api/v1/metadata/lookup?path=&schema=&table=   (control-plane proxy)
    → forwards to vastdb-query:8070
    → { rows, count, matched_by } | { error: "..." }
  Render inline in dialog; does NOT affect save state.
```

### 4.3 Invariants

- **Single write path.** Every mutation (toggle, edit, seed, seed-missing) goes through `saveMetadataPipelines(fullArray)` → `PUT /platform/settings`. No partial-update API exists.
- **Server validation authority.** `validatePipelineConfigList` enforces one-pipeline-per-`fileKind` and no-duplicate-extensions. Frontend does non-empty checks only; server is the regex authority.
- **Defaults single-source-of-truth.** `GET /pipelines/defaults` reads `default-pipelines.json` via `FileSeedLoader` and runs `validatePipelineConfigList` before returning.
- **Immutable `fileKind`.** Edit dialog renders `fileKind` as read-only. Changing identity would require delete+create, which is out of scope.

## 5. HTTP contracts

### 5.1 Reused (no change)

**`GET /api/v1/dataengine/pipelines/active[?force=true]`** — already shipped (`routes/dataengine-pipelines.ts:150`). Response:

```json
{
  "pipelines": [
    {
      "config": { "fileKind": "image", "functionName": "...", "extensions": [".exr"], "targetSchema": "frame_metadata", "targetTable": "files", "sidecarSchemaId": "frame@1", "displayLabel": "...", "enabled": true },
      "live": { "guid": "...", "name": "...", "description": "...", "owner": {...}, "createdAt": "...", "updatedAt": "...", "vrn": "...", "lastRevisionNumber": 5 },
      "status": "ok",
      "statusDetail": "optional human-readable diagnosis"
    }
  ]
}
```

**`PUT /api/v1/platform/settings`** — already shipped. Body accepts `{ dataEnginePipelines: DataEnginePipelineConfig[] }`. Validated server-side by `validatePipelineConfigList`. Returns 400 with `{ error: { code: "INVALID_PIPELINE_CONFIG", message: "pipelines[idx]: ..." } }` envelope on failure.

### 5.2 New

**`GET /api/v1/dataengine/pipelines/defaults`** — returns the canonical default pipeline list by re-reading the same seed JSON used by `pipeline-seed.ts`.

Request: no parameters.

Response 200:
```json
{
  "pipelines": [ /* DataEnginePipelineConfig[] — already normalized by validatePipelineConfigList */ ]
}
```

Response 500 (seed file missing, malformed, or failing validation):
```json
{ "error": { "code": "SEED_UNAVAILABLE", "message": "..." } }
```

Implementation: wrap `new FileSeedLoader().load()` + `validatePipelineConfigList`. Auth: same `admin:system_config` permission as other admin endpoints.

**`GET /api/v1/metadata/lookup?path=&schema=&table=`** — thin forwarder to `vastdb-query`.

Implementation: reuses the existing `proxyToVastdbQuery(path)` helper at `services/control-plane/src/routes/exr-metadata.ts:26`, which reads the vastdb-query base URL from `process.env.VASTDB_QUERY_URL` (Docker default `http://vastdb-query:8070`) and already handles the 503 fallback on fetch failure. The new route constructs the downstream path `/api/v1/metadata/lookup?path=…&schema=…&table=…` from the incoming query, delegates, and relays.

Request query params (all required):
- `path`: full S3 URL or bucket-prefixed path
- `schema`: target VastDB schema
- `table`: target VastDB table

Response 200: relays the vastdb-query response body as-is (`{ rows, count, matched_by }`).

Response 503:
```json
{ "error": { "code": "LOOKUP_UNREACHABLE", "message": "vastdb-query unreachable: ..." } }
```

Response 400 (missing query params): standard error envelope.

Auth: `admin:system_config`. Rate limit: none for now (trusted internal page; revisit if test-lookup gets embedded in a user-facing page).

## 7. Error handling

| Condition | UX |
|---|---|
| Fetch `/active` fails | Page-level error banner; last successful rows preserved; retry button re-fetches. |
| `PUT /settings` returns 400 `INVALID_PIPELINE_CONFIG` | Error envelope message rendered in dialog above buttons; dialog stays open, form state preserved. |
| `PUT /settings` returns 403 | "Not permitted" banner; dialog stays open. |
| `PUT /settings` 5xx | "Save failed: {message}"; dialog stays open. |
| Inline toggle PUT fails | Rollback optimistic state + page banner. |
| Test-lookup returns error body | Red inline result in dialog; does not block save; clears on next Run. |
| Test-lookup 503 (vastdb-query down) | "Lookup service unreachable — VAST Database may be down"; does not block save. |
| `GET /defaults` fails | Banner "Default pipeline set unavailable — contact deploy admin"; seed button disabled after first failure in-session. |
| Missing status pill data (`status` absent) | Neutral "?" badge, no tooltip. Should not happen — the backend always sets `status`. |

**No auto-retry** on any failure. Every retry is a deliberate click.

## 9. Risks & mitigations

| Risk | Mitigation |
|---|---|
| Operator accidentally overwrites a production config | Dialog shows diff-style before/after? — out of scope for v1. For now: server validation error envelope and the fact that `fileKind` is immutable limit damage. |
| Seed file in newer deploys drifts from `pipeline-config.ts` shape | `GET /defaults` runs `validatePipelineConfigList` → any drift returns 500, banner surfaces it before operator damage. |
| Test-lookup endpoint leaks as a user-facing abuse vector | Auth gate `admin:system_config` + proxy (no direct vastdb-query exposure). Adequate for v1. |
| `VASTDB_QUERY_URL` env misconfigured / vastdb-query down | Existing `proxyToVastdbQuery` helper returns 503 with detail; the new route wraps that as `LOOKUP_UNREACHABLE`. Operator fixes the env (redeploy) — not runtime-editable, which is consistent with the existing `/exr-metadata/*` and `/video-metadata/*` consumers. |
| Naming collision with existing `/pipeline/functions` PipelinesTab (VAST native pipelines) | Nav label "Metadata Pipelines" disambiguates; page URL `/automation/pipelines` is distinct from `/pipeline/functions`. Docs will link both. |
