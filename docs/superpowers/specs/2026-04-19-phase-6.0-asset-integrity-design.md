# Phase 6.0 — Asset Integrity & Ops Hygiene — Design

**Date:** 2026-04-19
**Branch name (proposed):** `feat/phase-6.0-asset-integrity`
**Cycle scope:** five foundation capabilities that harden SpaceHarbor's operational surface — standardized retry/timing infrastructure, typed runtime function configs with a reusable admin UI pattern, authoritative asset counters, and data-access contracts for two new DataEngine functions (hashing, keyframes).

## Goal

Ship the infrastructure foundation that later cycles will build on. Concretely:

1. Any control-plane code that makes an outbound call to VAST DataEngine, VMS, or a remote service can wrap that call in a single standardized helper that gives it retry-with-backoff, per-attempt timeout, structured timing logs, and a Prometheus counter — without each caller re-implementing the pattern.
2. Any admin surface that exposes a numeric, boolean, or string knob to operators can render it via one reusable component that enforces min/max/type constraints, shows last-edited-by attribution, and persists through a central typed-config table.
3. The Asset Browser's header surfaces authoritative catalog-wide counters (not virtualizer-slice counters), including integrity counters that light up when the DataEngine functions land.
4. Two DataEngine function contracts are authored by SpaceHarbor (`asset-hash-generator`, `asset-keyframe-extractor`) so the external team implementing them has unambiguous input/output schemas and idempotency requirements.

## Non-goals

- No cross-asset duplicate detection logic. Hashes are written per-asset; comparison happens in a future cycle.
- No `relationships` table, no graph writes, no version/syndication/GDPR/AI-detection surfaces.
- No subclip extraction, no transcript, no OCR, no vector search, no inference wiring.
- No migration of existing admin pages (Metadata Pipelines, Scanner Forwarder, IAM, Custom Fields, Triggers, Workflows, Webhooks, etc.) through the new `function_configs` table. Those pages keep their current backends. `function_configs` is the preferred backend for NEW admin settings only.
- No DataEngine function implementation. This repository authors the data contracts; the function code lives in the external DataEngine functions team's repository. Our cycle ships the consumers + contracts; the functions land on their own timeline.

No dependency on externally-hosted services beyond what SpaceHarbor already runs (VAST DB, Event Broker, existing DataEngine functions). No shared buckets, schemas, tables, or functions with any other project. Pattern names are justified from SpaceHarbor's own trajectory. |

## Architecture

Four layers with an explicit dependency order. Layers A and B can run in parallel. C depends on B's schema shape. D depends on A (SettingCard + function_configs) + C (stats endpoint + integrity reader).

```
Layer A — Infra primitives                Layer B — DataEngine contracts (docs)
  callWithRetryAndTiming() helper           asset-hash-generator
  [timing] log convention                   asset-keyframe-extractor
  retry_attempts_total counter              asset_integrity.hashes
  function_configs typed table              asset_integrity.keyframes
  function-configs-store (60s cache)        ownership: external team's repo
  one retrofit target (vast-function-fetcher's 401 retry)
         │                                         │
         └─────────────┬───────────────────────────┘
                       ▼
              Layer C — Backend readers
               GET /api/v1/assets/stats
               GET /api/v1/assets/:id/integrity
               GET/PUT /api/v1/function-configs/:scope[/:key]
               graceful fallback when asset_integrity.* are empty
                       │
                       ▼
              Layer D — UI
               <SettingCard> component (typed input + constraints + audit metadata)
               <KpiCounterStrip> on Assets list
               /admin/integrity settings page
               Integrity tab on Asset Detail panel
```

## Components & contracts

### Layer A — Infra primitives

#### `callWithRetryAndTiming(fn, opts)`

File: `services/control-plane/src/infra/retry-with-timing.ts`.

```typescript
export interface RetryOptions {
  /** Operation name — goes into [timing] log prefix and structured fields. */
  op: string;
  /** Number of attempts (default: 6). */
  maxAttempts?: number;
  /** Base backoff schedule in ms (default: [5000, 15000, 30000, 60000, 120000]). */
  backoffMs?: readonly number[];
  /** ±50% jitter (default: true). */
  jitter?: boolean;
  /** Per-attempt timeout ceiling in ms (default: none). */
  perAttemptTimeoutMs?: number;
  /** Predicate — false means stop retrying. Default retries on any Error. */
  shouldRetry?: (err: unknown, attempt: number) => boolean;
  /** AbortSignal for mid-backoff cancellation. */
  signal?: AbortSignal;
  /** Fastify logger or equivalent. */
  log: FastifyBaseLogger;
}

export class RetryExhaustedError extends Error {
  readonly attempts: number;
  readonly op: string;
  readonly cause: unknown;
}

export async function callWithRetryAndTiming<T>(
  fn: () => Promise<T>,
  opts: RetryOptions,
): Promise<T>;
```

**Log format (greppable CSV-ish):**

```
[timing] op=vast-function-fetch attempt=1 latency_ms=823 cumulative_ms=823 status=ok
[timing] op=vast-function-fetch attempt=2 latency_ms=1204 cumulative_ms=7031 status=retry reason=ECONNRESET
[timing] op=vast-function-fetch attempt=2 latency_ms=615 cumulative_ms=7646 status=ok
```

**Counter metric:** `retry_attempts_total{op, outcome}` incremented on every attempt. Labels: `op` (whatever caller passed), `outcome ∈ {ok, retry, non_retryable, exhausted, timeout}`. Exposed on the existing Prometheus `/metrics` endpoint (added in the previous cycle for `metadata_lookup_fallback_total` — same surface, no new plumbing).

**Jitter:** `backoffMs[i] * (0.5 + Math.random())` — ±50% around the base, matching the backoff randomization already used in `services/control-plane/src/iam/auth-plugin.ts`.

**Retrofit target (validation):** migrate the one-shot 401-refresh-and-retry pattern in `services/control-plane/src/data-engine/vast-function-fetcher.ts` onto `callWithRetryAndTiming` with `maxAttempts: 2`, `backoffMs: [0]`, and a `shouldRetry` that returns true only on HTTP 401. Proves the wrapper works outside the default backoff schedule. Existing behavior preserved byte-for-byte.

#### `function_configs` table + store

Migration: `services/control-plane/src/db/migrations/NNN_function_configs.sql`.

```sql
CREATE TABLE IF NOT EXISTS function_configs (
  scope          VARCHAR NOT NULL,
  key            VARCHAR NOT NULL,
  value_type     VARCHAR NOT NULL,   -- "int" | "float" | "bool" | "string" | "duration_seconds"
  value_json     VARCHAR NOT NULL,   -- JSON-encoded current value
  default_json   VARCHAR NOT NULL,   -- JSON-encoded factory default
  min_json       VARCHAR,            -- JSON-encoded min (numeric/duration only)
  max_json       VARCHAR,            -- JSON-encoded max (numeric/duration only)
  description    VARCHAR NOT NULL,   -- human-readable prose for the UI
  label          VARCHAR NOT NULL,   -- short UI label
  category       VARCHAR NOT NULL,   -- grouping within scope
  last_edited_by VARCHAR,
  last_edited_at TIMESTAMP,
  PRIMARY KEY (scope, key)
);
```

Seed (idempotent) rows for scope=`asset-integrity`:

| key | type | default | min | max | label | category |
|---|---|---|---|---|---|---|
| `hash_timeout_seconds` | duration_seconds | 120 | 10 | 600 | Hash generation timeout | Hashing |
| `hash_concurrency_max` | int | 4 | 1 | 16 | Max concurrent hash jobs | Hashing |
| `perceptual_hash_enabled` | bool | true | — | — | Perceptual hash enabled | Hashing |
| `keyframe_count` | int | 10 | 1 | 30 | Keyframes per video | Keyframes |
| `keyframe_extraction_timeout_seconds` | duration_seconds | 180 | 30 | 900 | Keyframe extraction timeout | Keyframes |
| `keyframe_extraction_enabled` | bool | true | — | — | Keyframe extraction enabled | Keyframes |

Store: `services/control-plane/src/config/function-configs-store.ts`.

```typescript
export interface FunctionConfig {
  scope: string;
  key: string;
  value: unknown;         // typed-decoded from value_json
  valueType: "int" | "float" | "bool" | "string" | "duration_seconds";
  default: unknown;
  min?: number;
  max?: number;
  description: string;
  label: string;
  category: string;
  lastEditedBy: string | null;
  lastEditedAt: string | null;
}

export async function getScope(scope: string): Promise<FunctionConfig[]>;
export async function getValue<T>(scope: string, key: string): Promise<T>;
export async function setValue(scope: string, key: string, value: unknown, actor: string): Promise<void>;
export async function resetToDefault(scope: string, key: string, actor: string): Promise<void>;
```

Module-level cache: `Map<string, { data: FunctionConfig[], expiresAt: number }>` keyed by scope, 60s TTL. Same shape as `useDataEnginePipelines.ts`'s module cache. `setValue` invalidates the cache entry synchronously before returning; read-after-write is consistent.

### Layer B — DataEngine contract specs (docs-only)

These files author the data contract that the external DataEngine functions team consumes. No code changes in this repository — the functions are implemented in their repository on their schedule. Our Layer C readers ship with graceful fallback so that absent data is not an error.

#### `docs/contracts/asset-hash-generator.md`

- **Trigger:** Kafka topic `spaceharbor.assets.ingested`, published by `services/control-plane/src/routes/scanner-ingest.ts` when a new asset lands in a watched bucket. No S3-event dependency; the function consumes our broker.
- **Input envelope:** `{ assetId, sourceUri, bucket, key, sizeBytes, mime?, fileKind, ingestedAt }`.
- **Output table:** `asset_integrity.hashes`.

```
asset_integrity.hashes
  asset_id           VARCHAR   PRIMARY KEY
  source_uri         VARCHAR   NOT NULL
  sha256             VARCHAR(64)
  perceptual_hash    VARCHAR             -- nullable; only set for image/video kinds
  algorithm_version  VARCHAR             -- e.g. "pdqhash-1.0"
  bytes_hashed       BIGINT
  hashed_at          TIMESTAMP
  hash_duration_ms   BIGINT
```

- **Idempotency:** upsert keyed by `asset_id`. Re-running on the same asset produces the same row; `algorithm_version` and `hashed_at` reflect the most recent run.
- **Failure reporting:** the function must not write a partial row. If SHA-256 computes successfully but perceptual hashing fails, the row is either written fully (with `perceptual_hash = null`) or not at all — never with a stale mix of fields.
- **Tunable knobs (read via `function_configs` scope `asset-integrity`):** `hash_timeout_seconds`, `hash_concurrency_max`, `perceptual_hash_enabled`.

#### `docs/contracts/asset-keyframe-extractor.md`

- **Trigger:** same topic `spaceharbor.assets.ingested`; the function filters internally to `fileKind ∈ {video, raw_camera}` (image kinds are skipped).
- **Input envelope:** same as hash-generator.
- **Output object side-effect:** writes up to `keyframe_count` JPEG frames into `<bucket>/.proxies/<asset_id>/keyframes/NNNN.jpg` with stable names.
- **Output table:** `asset_integrity.keyframes`.

```
asset_integrity.keyframes
  asset_id                 VARCHAR   PRIMARY KEY
  source_uri               VARCHAR   NOT NULL
  keyframe_count           INT
  keyframe_prefix          VARCHAR             -- e.g. "sergio-spaceharbor/.proxies/abc123/keyframes/"
  thumbnail_key            VARCHAR             -- one frame chosen for grid display
  extracted_at             TIMESTAMP
  extraction_duration_ms   BIGINT
```

- **Idempotency:** upsert keyed by `asset_id`; re-running overwrites JPEGs under the same deterministic names.
- **Tunable knobs (read via `function_configs` scope `asset-integrity`):** `keyframe_count`, `keyframe_extraction_timeout_seconds`, `keyframe_extraction_enabled`.

**Ownership note (included in both contract docs):** This contract is authored by SpaceHarbor. The function implementation lives in the DataEngine functions team's repository. This contract depends only on infrastructure SpaceHarbor runs directly (VAST DB, Event Broker, media buckets). No assumption is made about any other project's schemas, tables, buckets, functions, or runtime endpoints.

### Layer C — Backend readers

#### `GET /api/v1/assets/stats`

File: `services/control-plane/src/routes/asset-stats.ts`.

**Response 200:**

```json
{
  "total": 1234,
  "byStatus": { "pending": 12, "in_pipeline": 3, "processed": 1200, "failed": 19 },
  "byKind": { "image": 800, "video": 400, "raw_camera": 30, "other": 4 },
  "integrity": { "hashed": 0, "with_keyframes": 0 }

```

- Counts aggregated from the asset catalog table + LEFT JOIN on `asset_integrity.hashes` and `asset_integrity.keyframes`.
- Graceful when integrity tables are empty: both integrity counters return `0`.
- Requires no new IAM permission — same access rules as `GET /assets`.
- No caching at this layer (small aggregate query; adding cache is a later optimization if needed).

#### `GET /api/v1/assets/:id/integrity`

File: `services/control-plane/src/routes/asset-integrity.ts`.

**Response 200:**

```json
{
  "assetId": "abc123",
  "sources": { "hashes": "ok", "keyframes": "empty" },

#### `GET /api/v1/function-configs/:scope` and `PUT /api/v1/function-configs/:scope/:key`

File: `services/control-plane/src/routes/function-configs.ts`.

- **GET** returns the full scope shape (array of `FunctionConfig` objects). Gated by `admin:system_config` IAM permission.
- **PUT** writes a new value for a single key. Validates against `value_type` + `min` + `max` declared at seed time. Returns the updated row. Gated by `admin:system_config`. Write goes through the audit-chain helper at `services/control-plane/src/infra/audit-chain.ts` (same pattern as platform settings).
- No DELETE — settings are declared at seed time, not user-managed.

### Ingest flow (forward-looking; activates when the DataEngine functions land)

```
User/scanner drops file into watched S3 bucket
  ↓
Scanner Forwarder (services/scanner-function/) — HMAC-signed POST → control-plane
  ↓
control-plane /scanner/ingest publishes to Kafka: spaceharbor.assets.ingested
  ↓

  ├── existing consumers: frame-metadata-extractor, video-metadata-extractor,

### Admin settings flow

```
Operator navigates to /admin/integrity
  ↓
useFunctionConfigs("asset-integrity")
  ├── cache hit: render immediately
  └── cache miss: GET /api/v1/function-configs/asset-integrity
                   → control-plane reads function_configs table (60s module cache)
                   → returns FunctionConfig[]
  ↓
IntegritySettingsPage groups by category → <SettingCard> per config
  ↓

```

DataEngine functions on their next invocation read the new value through their own 60s cache. No restart required.

### Signal paths

- **`[timing]` log prefix** fires on every `callWithRetryAndTiming` call site. The initial retrofit at `vast-function-fetcher.ts` emits lines on every VAST function fetch. Ops can grep `[timing] op=vast-function-fetch` for latency distribution. The convention is documented in `services/control-plane/ARCHITECTURE.md`.
- **Counter metric** `retry_attempts_total{op, outcome}` increments on every attempt. Exposed on the existing `/metrics` endpoint.
- **Audit writes** — every `function_configs` write is captured through the existing `audit-chain.ts` infrastructure.

## Error handling

### Layer A — `callWithRetryAndTiming`

| Scenario | Behavior | Log | Metric |
|---|---|---|---|
| First attempt succeeds | Return result | 1 line `status=ok, attempt=1` | `outcome="ok"` |
| Attempt fails, retries remain | Sleep `backoffMs[i] ± jitter`, try again | `status=retry, reason=<err>` | `outcome="retry"` |
| `shouldRetry` returns false | Throw immediately | `status=error, reason=<err>` | `outcome="non_retryable"` |
| All attempts exhausted | Throw `RetryExhaustedError` wrapping last cause | `status=exhausted` | `outcome="exhausted"` |
| `perAttemptTimeoutMs` fires | `AbortError` thrown; counted as retryable unless `shouldRetry` says no | `status=timeout` | `outcome="timeout"` |

Invariants: backoff sleeps honor an external `AbortSignal`; jitter is ±50%; `RetryExhaustedError` preserves `cause` via ES2022 `Error.cause`.

### Layer A — `function_configs` error model

| Scenario | Response |
|---|---|
| `GET /function-configs/:scope` — scope has no rows | `200 { configs: [] }` |
| `PUT` — key not defined in scope | `404 { code: "CONFIG_KEY_NOT_FOUND" }` |
| `PUT` — value fails type check | `400 { code: "VALIDATION_ERROR", message: "value must be integer" }` |
| `PUT` — value below min or above max | `400 { code: "VALIDATION_ERROR", message: "value must be <= 600" }` |
| `PUT` — caller lacks `admin:system_config` | `403 { code: "FORBIDDEN" }` |
| `PUT` — DB write fails | `503 { code: "DB_UNREACHABLE" }` |
| Read-after-write consistency | Cache invalidation is synchronous with the write, inside the same transaction. |

### Layer B — Contract invariants (enforced by review, not code)

- Output idempotency: repeated execution on the same `asset_id` produces stable output (upsert keyed by `asset_id`; deterministic JPEG names).
- No destructive writes: the functions NEVER delete rows in `asset_integrity.*` or objects under `<bucket>/.proxies/<asset_id>/`.
- No cross-asset mutations: hash-generator writes only its own asset's row; it never writes to `relationships` or similar (which don't exist in this cycle).
- Failure reporting: functions must not write partial rows.

### Layer C — Reader route error surfaces

| Route | Scenario | Response |
|---|---|---|
| `GET /assets/stats` | Catalog read fails | `503 DB_UNREACHABLE` |
| `GET /assets/stats` | Integrity join empty (tables exist, no data) | `200` with `integrity: { hashed: 0, with_keyframes: 0 }` |
| `GET /assets/:id/integrity` | Asset not in catalog | `404 ASSET_NOT_FOUND` |
| `GET /assets/:id/integrity` | Asset exists, neither integrity row exists | `200` with `sources: { hashes: "empty", keyframes: "empty" }`, both objects `null` |
| `GET /assets/:id/integrity` | Hashes only | `sources: { hashes: "ok", keyframes: "empty" }` |
| `GET /assets/:id/integrity` | DB read throws | `503 DB_UNREACHABLE` |

### Layer D — UI error surfaces

| Condition | UI behavior |
|---|---|
| `useAssetStats` throws | Each counter shows `—` + warning icon; asset list unaffected |
| `useAssetIntegrity` throws | Integrity tab shows "Unable to load integrity data" + Retry button |

## Risks & mitigations

| Risk | Mitigation |
|---|---|
| DataEngine functions slip; integrity data never populates. | Layer C readers gracefully return empty; Layer D UI renders empty-states. Phase 6.0 ships useful infrastructure even if the functions take longer. |
| Existing admin pages drift from the new `function_configs` pattern forever. | Accepted for this cycle. Migration is a separate future cycle. Document the intent in `services/control-plane/ARCHITECTURE.md`. |
| Jitter test is statistical and could flake. | Run 100 iterations; assert mean + stddev within a generous band (not exact values). |
| `retry_attempts_total` counter cardinality explodes if `op` labels proliferate. | Callers are encouraged to pass stable operation names from a small set; the label is caller-owned, so misuse is catchable at code-review time. Not a runtime issue. |
| Migration to new `function_configs` table fails on existing deployments. | Migration is `CREATE TABLE IF NOT EXISTS` + idempotent seed rows; safe to re-run. |
| Audit-chain writes on `setValue` slow down the Save path. | Accepted: admin knobs are edited infrequently; audit is more valuable than latency. |
