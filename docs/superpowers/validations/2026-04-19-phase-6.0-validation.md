## Post-validation hardening (after initial validation pass)

Three follow-up commits landed after the initial validation report and address V1 follow-ups identified during ship review. All are functional safety improvements with their own tests; none alter the core Phase 6.0 surface area.

- **`ba2c454`** — Adds JSON Schema `pattern: ^[A-Za-z0-9_.-]+$` (with `minLength: 1`, `maxLength: 128`) to the `:id` param on `GET /api/v1/assets/:id/integrity`. Defends against any potential SQL string interpolation in `vast-persistence.getAssetIntegrity` if a future caller bypasses route validation. Adds 1 regression test (`malformed asset id: 400`).
- **`517be29`** — Adds `notImplemented?: boolean` flag on `FunctionConfigsDeps`. When true, both `GET /function-configs/:scope` and `PUT /function-configs/:scope/:key` short-circuit with 503 `NOT_IMPLEMENTED` after the admin gate. `app.ts` sets the flag while the in-memory `StoreDeps` stub is in place — admins editing seeded keys get an unambiguous "not yet wired" signal instead of a misleading 404 `CONFIG_KEY_NOT_FOUND`. Removes follow-up #1 from the original Follow-ups list. Adds 3 regression tests.
- **Persistence-layer assetId guard** — Mirrors the route-level pattern in `VastPersistenceAdapter.getAssetIntegrity`. The method now early-returns `{assetExists: false, hashes: null, keyframes: null}` if `assetId` fails `^[A-Za-z0-9_.-]+$` or exceeds 128 chars, without building or executing any SQL. Closes the gap that the `ba2c454` route guard left open for non-route callers (no such callers exist today; this is belt-and-suspenders). Adds 1 regression test in `vast-persistence-sql.test.ts` asserting Trino is never reached for 7 representative malformed-ID payloads.
